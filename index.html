Quick Start

Running locally

MindMesh uses a custom Node server — npm run dev starts server.js directly, which boots Next.js and attaches Socket.IO to the same HTTP server.

# Clone & install git clone https://github.com/imjoe77/Mind-Mesh.git cd Mind-Mesh && npm install # Configure environment cp .env.example .env.local # Fill in all required variables (see Environment section) # Start — runs server.js, not next dev npm run dev
Note: The dev script runs node server.js, not the standard Next.js dev server. This is required so Socket.IO can attach to the HTTP server alongside Next.js.

Real-Time Layer

WebSockets —
how Socket.IO is used

Socket.IO is the central nervous system of the platform. A custom server.js wraps Next.js and attaches a persistent Socket.IO server. Every collaborative feature — whiteboard, pomodoro, chat, AI sync, and feature locking — runs through this event bus.

Feature locking: When a user opens the AI Tutor or PDF panel, a feature-lock event is emitted. The server broadcasts the lock state to all peers — others see a "currently in use" notice. When the user is done, feature-unlock releases it.
Peer-to-Peer Video

WebRTC — how the
video mesh works

Video data never touches the server. Socket.IO only carries the signaling handshake — a few kilobytes of SDP and ICE packets. After that, the video stream travels directly browser-to-browser.

WebRTC Signaling Flow
P2P Active
👤
Peer A
Initiator
① SDP Offer
Socket.IO
Signal relay only
② SDP Answer
👤
Peer B
Responder
③ ICE candidates exchanged via STUN → Direct P2P stream ✓
1
Peer A calls getUserMedia() — camera/mic stream captured locally
2
Creates RTCPeerConnection with Google STUN servers in ICE config
3
createOffer() → set local description → emit webrtc-offer via Socket.IO
4
Peer B receives offer, sets remote description, calls createAnswer()
5
ICE candidates exchanged via webrtc-ice-candidate — STUN resolves public IPs
6
Direct P2P video stream established. Server no longer involved in video data.
Why P2P? Video never touches the server — no server bandwidth cost, lower latency. The server only relays the handshake (a few KB total). After that, streams go straight between browsers.

Deployment

Hosting strategy

Both the frontend and backend are deployed on Render as a single unified service. Because MindMesh uses a custom server.js that attaches Socket.IO to the Next.js HTTP server, the entire app runs as one persistent Node process — no split deployment needed.

🎨Render — Full Stack

One service runs everything — Next.js frontend, API routes, and Socket.IO WebSocket server all from a single server.js process.

Buildnpm install && npm run build
StartNODE_ENV=production node server.js
Health/health → 200 OK
Livemind-mesh-refactored.onrender.com
⚙️Why Render over Vercel

Vercel's serverless runtime kills long-lived connections after a few seconds — Socket.IO WebSockets would disconnect immediately. Render runs a persistent process, keeping WebSocket connections alive for the entire session.

Socket.IOPersistent ✓
WebRTC SignalingPersistent ✓

Single service advantage: No CORS configuration, no API proxying, no split environment variables. The frontend, API, and WebSocket server all share the same origin — mind-mesh-refactored.onrender.com.
